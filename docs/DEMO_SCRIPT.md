# ProofMesh Demo Recording Script

## Recording target

- Length: 2 minutes 30 seconds to 3 minutes
- Viewport: desktop browser at 1440 x 900 or larger
- Language: English narration is recommended for Demo Day
- Input: use the preloaded Great Wall claim for the stable rehearsal
- Before recording: close unrelated tabs, hide bookmarks, disable notifications, and reload ProofMesh to the input screen

## One-take flow and narration

### 0:00-0:18 - The problem

**On screen:** Keep the full ProofMesh input workspace visible.

**Say:**

> Every day, a viral claim reaches us before the evidence does. Most AI tools return another answer. ProofMesh returns an auditable case.

### 0:18-0:38 - The product promise

**On screen:** Point to Article, X post, Claim, and Image, then the three capability chips.

**Say:**

> I can submit an article, an X post, a text claim, or an image. ProofMesh treats the input as untrusted, separates it into testable claims, and keeps every source, model, and request ID visible.

### 0:38-0:48 - Start with one keystroke

**On screen:** Keep the preloaded Great Wall claim and press Enter.

**Say:**

> Let us test a familiar claim: that the Great Wall of China is visible from the Moon with the naked eye.

### 0:48-1:22 - Agent orchestration

**On screen:** Stay on the animated Agent workspace. Let Claim Mapper, Investigator, Challenger, and Consensus activate in sequence.

**Say:**

> The page now becomes a live case engine. Kimi maps the claim and gathers evidence. A second Kimi role builds the strongest evidence-grounded assessment. MiniMax challenges the assumptions and looks for missing context. Finally, a cross-model judge reconciles the disagreement. These are not decorative loading states: the task is moving through four real inference stages on GonkaRouter.

### 1:22-1:48 - The verdict arrives

**On screen:** Let ProofMesh enter the Result workspace automatically. Pause on Overview with the score, verdict, claim map, timeline, and reversal condition visible.

**Say:**

> The result arrives as a workspace, not a wall of text. This claim scores zero, is marked false with calibrated confidence, and is split into independently testable statements. The timeline shows how the score evolved, while the reversal condition states what evidence could change the verdict.

### 1:48-2:08 - Evidence

**On screen:** Open **Evidence**.

**Say:**

> The Evidence panel preserves provenance for every source and makes source diversity visible. Reviewers can inspect the exact packet that the agents received instead of trusting a hidden retrieval step.

### 2:08-2:28 - Model disagreement

**On screen:** Open **Agents**.

**Say:**

> The Agents panel exposes the Investigator, Challenger, and Consensus side by side. Their confidence, reasoning, model identity, and request IDs remain attached to the case.

### 2:28-2:47 - Audit layer

**On screen:** Open **Audit**.

**Say:**

> The Audit panel signs the final consensus with a complete inference ledger. ProofMesh is not another AI answer. It is a transparent verification workflow that can be inspected and repeated.

### 2:47-3:00 - Shareable output

**On screen:** Click **Download Proof Card**.

**Say:**

> The case exports as a Proof Card for social media, moderation, or newsroom workflows. ProofMesh makes the process of reaching truth visible.

## Recording acceptance checklist

- The opening frame shows the full headline and input terminal.
- Pressing Enter visibly changes the application to the Agent workspace.
- At least three Agent nodes and the four-stage tracker appear.
- The result opens automatically without scrolling.
- The Overview frame shows the verdict and Truth Score together.
- Evidence, Agents, and Audit tabs are each opened once.
- At least two source names and three real Gonka request IDs are legible.
- The reversal condition appears in Overview.
- Proof Card export succeeds.
- No API key, email address, account menu, or private notification appears.

## Backup plan

If the live API is unavailable during recording, use the clearly labeled curated demo simulation. Do not describe demo request IDs as live network calls. The competition recording should use a successful live Gonka run whenever possible.
